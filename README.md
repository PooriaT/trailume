# Trailume

Trailume is a local-first training recap app. It pulls activity data from Strava, computes deterministic insights, and renders a recap dashboard with optional narrative text generated by a local Ollama model.

## Product overview

Trailume answers a focused question: **"How did this training block go?"**

MVP user flow:
1. Authenticate with Strava.
2. Select a date range and activity type.
3. Review fetched activities.
4. Generate a recap from deterministic analytics.
5. Add narrative text from Ollama when available (otherwise deterministic fallback).

## Architecture summary

Trailume is a monorepo with two runnable apps:

- **`apps/web`**: Next.js UI for auth flow, recap controls, and recap presentation.
- **`apps/api`**: FastAPI service for Strava integration, analytics computation, and narrative generation.

Current backend boundaries:
- `services/strava`: OAuth, token refresh, and activity fetch/normalize.
- `services/analytics`: deterministic metrics and insight generation.
- `services/narrative`: provider contract, Ollama provider, deterministic fallback provider.
- `api/routes`: HTTP I/O, validation, session checks, and response mapping.

## Local setup

### Prerequisites

- Node.js 20+
- pnpm 9+
- Python 3.13
- Poetry 1.8+
- Optional: Ollama (for local LLM narrative generation)

### 1) Create env files

Copy the example env files:

```bash
cp .env.example apps/web/.env.local
cp apps/api/.env.example apps/api/.env
```

### 2) Install dependencies

From repository root:

```bash
pnpm install
cd apps/api
poetry install
cd ../..
```

### 3) Run frontend and backend

From repository root:

```bash
pnpm dev
```

Or run each app separately:

```bash
pnpm dev:web
pnpm dev:api
```

Default URLs:
- Frontend: `http://localhost:3000`
- API OpenAPI docs: `http://localhost:8000/docs`

## Environment variables

### Frontend (`apps/web/.env.local`)

- `NEXT_PUBLIC_API_BASE_URL` (default: `http://localhost:8000`)

### Backend (`apps/api/.env`)

Core runtime:
- `APP_ENV`
- `API_HOST`
- `API_PORT`
- `WEB_APP_URL`

Strava integration:
- `STRAVA_CLIENT_ID`
- `STRAVA_CLIENT_SECRET`
- `STRAVA_REDIRECT_URI` (default: `http://localhost:8000/api/v1/auth/strava/callback`)

Narrative/Ollama:
- `OLLAMA_BASE_URL` (default: `http://localhost:11434`)
- `OLLAMA_MODEL` (default: `gemma4`)
- `OLLAMA_TIMEOUT_SECONDS`
- `OLLAMA_TEMPERATURE`
- `OLLAMA_TOP_P`
- `OLLAMA_NUM_PREDICT`

Session cookie behavior:
- `SESSION_COOKIE_SECURE`
- `SESSION_COOKIE_SAMESITE`

## Strava setup

1. Open `https://www.strava.com/settings/api` and create an API application.
2. Use `localhost` as callback domain for local development.
3. Configure backend env values:
   - `STRAVA_CLIENT_ID`
   - `STRAVA_CLIENT_SECRET`
   - `STRAVA_REDIRECT_URI=http://localhost:8000/api/v1/auth/strava/callback`
4. Start Trailume and use **Connect with Strava** from the home page.

## Ollama setup

Install Ollama from `https://ollama.com/download`, then run:

```bash
ollama serve
ollama pull gemma4
```

If you use another local model, set `OLLAMA_MODEL` in `apps/api/.env`.

Narrative provider behavior:
- If Ollama is reachable and the configured model exists, Trailume uses Ollama.
- Otherwise Trailume automatically falls back to deterministic narrative generation.

## Backend developer workflow (Poetry)

All backend commands are Poetry-based:

```bash
cd apps/api
poetry run uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
poetry run pytest
poetry run ruff check app tests
poetry run ruff format app tests
```

Root scripts (`pnpm dev:api`, `pnpm test:api`, `pnpm lint:api`, `pnpm format:api`) now call these Poetry commands.

## Current MVP scope

Included:
- Strava OAuth login/callback/status/logout flow.
- Session-backed Strava activity fetch by date range and type.
- Deterministic analytics output (summary metrics, highlights, trend series, standout activities, insight flags).
- Narrative provider boundary with Ollama primary + deterministic fallback.
- Recap UI rendering for generated payload sections.

Not included:
- Persistent token/session storage beyond process memory.
- Background jobs or queued recap generation.
- Production deployment hardening and multi-user account model.
- Fully wired route geometry visualization.

## Known limitations

- API session/token state is in-memory and resets when the API process restarts.
- Strava rate-limit handling is minimal and MVP-oriented.
- Narrative quality depends on local Ollama model availability/quality.
- Shared schema package exists but cross-app schema enforcement is still mostly manual.

## Roadmap

1. **Recap depth**: stronger scene-level insights and richer training progression analysis.
2. **Visualization**: map overlays, repeated-route views, and better standout activity context.
3. **Comparisons**: period-over-period recap mode.
4. **Provider expansion**: additional activity providers beyond Strava while preserving normalized contracts.
5. **Narrative portability**: keep provider abstraction while enabling cloud-model backends later.
